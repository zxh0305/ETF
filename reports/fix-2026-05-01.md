# ETF数据质量修复报告（2026-05-01）

## 目标
修复ETF估值补全系统的数据质量问题，特别是申万行业加权PE计算错误和cninfo LOF误映射。

## 发现的三个核心问题

### 问题1：申万加权PE公式错误（根本原因）
旧公式：`Σ(PE_i × NW_i) / pos_nw_sum × total_weight`  
其中NW_i = w_i/total_weight，pos_nw_sum = Σ(NW_i) ≈ 1

实际执行过程：  
- `weighted_pe = round(sum(pe * nw for pe, _, _, _, nw in pos_entries) / pos_nw_sum, 2)`  
- → 变成 `Σ(PE_i × NW_i) / pos_nw_sum`，pos_nw_sum ≈ 1  
- 再 `× total_weight / pos_nw_sum` → 变成 `Σ(PE_i × NW_i) / pos_nw_sum × total_weight / pos_nw_sum`

分子分母都是pos_nw_sum，约等于1，但乘了两次total_weight，导致PE被严重压缩。

**正确公式**：`weighted_pe = Σ(PE_i × w_i) / Σ(w_i for PE_i>0)`  
- 手工验证：食品饮料PE=1700.29/82.82=20.53（与市场一致）
- 旧代码：1700.36/1.0000≈1700（错误）

### 问题2：cninfo LOF fallback太宽泛
`CNINFO_KW_MAP`中 `"LOF": "C05"` 导致201只LOF全被映射到"主要消费(PE=20.32)"。

包括：石油LOF（sz162719）、原油LOF（sh501018）、商品LOF、华宝油气等，全部错误。

### 问题3：申万行业成分数据源问题
akshare的`index_component_sw`返回的申万行业成分与标准分类不一致：
- 801710"银行"实际包含建材/水泥股（北新建材、海螺水泥等）
- 导致银行PE=65.81而非预期的5-10

## 实施的修复

### 修复1：申万加权PE公式（sw_industry_valuation.py + sw_industry_valuation_latest.json）
```python
# 正确公式
pos_entries = [(pe,pb,c,w,nw) for pe,pb,c,w,nw in valid_entries if pe > 0]
pos_nw_sum = sum(nw for _,_,_,_,nw in pos_entries)
weighted_pe = round(sum(pe * nw for pe,_,_,_,nw in pos_entries) * total_weight / pos_nw_sum, 2)
```
同步手工修正sw_industry_valuation_latest.json中26个行业的weighted_pe值。

### 修复2：cninfo LOF fallback（etf_valuation_enricher.py）
- 删除 `"LOF": "C05", "定开": "C05"`（主要消费）
- 改为 `"LOF": "C03", "定开": "C03"`（工业，均值PE=27.7，标注为fallback）
- 新增精确关键词：科技→C08、港股→C07、债券→C03、商品→C01、纳斯达克→C08等

### 修复3：分位估算函数重校
```python
def estimate_pe_percentile(pe):
    if pe < 10: return 10   # 历史极低
    elif pe < 15: return 20
    elif pe < 20: return 35
    elif pe < 25: return 50
    elif pe < 30: return 65
    elif pe < 40: return 78
    else: return 90
```

## 遗留问题

1. **申万部分行业成分错误**：东财API成分与标准申万分类不一致，导致银行/钢铁/电子等行业PE偏高。需修复sw_industry_valuation.py中的成分获取逻辑，使用正确的申万行业列表API。
2. **科创50/创业板指分位缺失**：乐咕乐股不支持这两个指数，相关ETF暂映射到创业板50(399673)替代。
3. **真实历史分位仅47只ETF**：乐咕乐股仅支持12个宽基指数，其余335只ETF无法获得真实分位。
